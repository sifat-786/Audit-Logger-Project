  \LARGE \textbf{Iris: An Observable and Auditable Governance Platform for Tool-Agnostic AI Agents}
\end{center}
\vspace{0.4cm}

\begin{abstract}
Autonomous AI agents utilizing the ReAct (Reasoning and Action) loop paradigm pose substantial security and predictability risks when executing actions in production environments. These include infinite reasoning loops, high token latency, non-deterministic execution paths, and lack of system-level audit visibility. This paper introduces \textbf{Iris}, a governance-aware framework that enforces safety boundaries around tool-agnostic AI agents. By replacing open-loop autonomous reasoning with a deterministic \textit{Controlled Intent Routing} pattern, Iris guarantees single-turn executions, intercepts unauthorized tool invocations through a granular policy matrix, and generates a structured, thread-safe, and immutable multi-format audit trail. System verification against a 70-scenario catalogue demonstrates 100\% compliance in risk classification and security policy enforcement.
\end{abstract}

\section{Introduction}

\subsection{Background and Context}
In recent years, the deployment of Large Language Models (LLMs) as autonomous agents has transitioned from academic research to enterprise applications. Modern AI agents are increasingly designed to be tool-agnostic, interacting directly with operating systems, databases, cloud resources, and external communication APIs. Typically, these agents are implemented using autonomous loops, such as the Reasoning and Action (ReAct) paradigm. Under the ReAct model, the agent receives a user prompt, executes a series of inner monologues, selects tools, parses outputs, and recursively iterates until it determines the target objective has been achieved. 

While the ReAct paradigm offers remarkable flexibility in addressing open-ended questions, its open-loop nature presents critical operational risks in production networks. Since the agent determines its own actions dynamically at runtime, system administrators cannot predict the execution path. Furthermore, standard logging frameworks are often insufficient for capture-and-recovery auditing, capturing only superficial model outputs without structured tracing metadata.

\subsection{Problem Statement}
The core vulnerabilities in deploying autonomous agent loops in enterprise settings can be summarized in four distinct vectors:
\begin{enumerate}
    \item \textbf{Infinite Execution Loops}: Language models can enter repetitive execution loops if a tool returns unexpected formatting or minor errors. This causes high token latency and severe compute cost accumulation.
    \item \textbf{State Mutation Non-Determinism}: Agents have direct access to write file systems, update databases, and send emails. A misclassified tool parameter can lead to irreversible state corruption or unintended external communications.
    \item \textbf{Telemetry Deficit}: Audits require tracing the complete request lifecycle, including intent classification, tool inputs, data scopes, risk tiers, and execution outcomes. Standard logging outputs fail to provide structured databases for query auditing.
    \item \textbf{Absence of Security Guardrails}: Standard architectures execute all tool invocations with uniform privilege levels. Low-risk operations (e.g., calculations) are treated with the same validation path as critical-risk tasks (e.g., executing terminal command scripts).
\end{enumerate}

\subsection{Proposed Solution: Iris}
To resolve these security vulnerabilities, we present \textbf{Iris}, a governance-first observability platform. Iris wraps tool-agnostic agents in a deterministic \textit{Controlled Intent Routing} pattern. This system intercepts proposed actions prior to execution, evaluates them against a risk policy matrix, logs all states to a thread-safe multi-format data store, and presents a visual observability dashboard.

\section{Related Work}
Existing guardrail frameworks for language models, such as LlamaGuard, Guardrails AI, and NeMo Guardrails, focus primarily on prompt filtering and output validation. While effective for content moderation, they lack native mechanisms for state-mutating tool execution. In contrast, multi-agent frameworks like LangGraph and AutoGen provide routing state machines but default to autonomous loops. Iris fills this gap by separating intent extraction from tool execution, introducing a deterministic policy engine that acts as an intercepting proxy before any system action occurs.

\subsection{Component Descriptions}
\begin{itemize}
    \item \textbf{User Interface Layer}: Implemented via Streamlit, injecting a custom CSS system (\texttt{style.css}) to render chat bubbles, system metrics, tool lists, and execution trace panels.
    \item \textbf{Controlled Intent Router}: The orchestrator that processes input queries, binds tool definitions, and evaluates the LLM's parsed intent. It executes exactly one turn, preventing execution loops.
    \item \textbf{Governance and Safety Engine}: Evaluates the proposed tool call using the policy and classification engines, returning approval decisions.
    \item \textbf{Tool Execution Layer}: Executes approved actions against local systems or remote APIs. It includes an asynchronous scheduler that spawns background worker threads.
    \item \textbf{Telemetry and Auditing Sink}: Writes structured logs to memory buffers and disk databases (JSON, CSV, SQLite) using thread-safe synchronization locks.
\end{itemize}

\subsection{Request Lifecycle}
A typical request progresses through the following stages:
\begin{enumerate}
    \item \textbf{Ingress}: The user types a query (e.g., \textit{``Send an email to user@example.com containing report.txt''}) in the text input box.
    \item \textbf{Validation check}: \texttt{query\_validator.py} runs regex-based checks to detect domain spelling errors (e.g., \texttt{gmal.com}) or keyword typos, returning correction warnings.
    \item \textbf{LLM Intent Parsing}: The query is sent to the LLM (Llama 3.3) with tool schemas bound. The LLM extracts the target tool (\texttt{send\_email}) and arguments (\texttt{to="user@example.com"}, \texttt{body="..."}).
    \item \textbf{Governance Interception}: The router passes the tool name to \texttt{risk\_classifier.py}, which maps it to \texttt{HIGH} risk. \texttt{decision\_engine.py} flags it as \texttt{need approval}.
    \item \textbf{Audit Logging (Pending State)}: The logger writes a record to the databases with status \texttt{SKIPPED} and decision \texttt{need approval}.
    \item \textbf{UI Render (Halted state)}: The dashboard displays the pending trace and provides an ``Approve \& Execute'' button.
    \item \textbf{Manual Approval Execution}: When clicked, the dashboard bypasses the routing check, executes the tool, passes the results to the LLM to write a natural language summary, logs the status as \texttt{SUCCESS}, and updates the UI.
\end{enumerate}

\subsection{Detailed Workflow Step Explanations}
\begin{enumerate}
    \item \textbf{Input Analysis}: User query input is processed by regex spellcheckers.
    \item \textbf{LLM Tool Binding Check}: If no tool call is generated by the LLM, the intent is classified as \texttt{NORMAL\_CHAT}. The flow writes an audit log and renders the response.
    \item \textbf{Risk Level Determination}: If a tool call is generated, the tool name is checked against the static classification map.
    \item \textbf{Policy Decision Path}:
    \begin{itemize}
        \item \textbf{\texttt{CRITICAL} Risk}: The action (e.g., \texttt{execute\_terminal\_command}) is blocked immediately. The status is set to \texttt{SKIPPED}, the decision is marked as \texttt{reject}, and a block notification is displayed.
        \item \textbf{\texttt{HIGH} Risk}: The action (e.g., \texttt{send\_email}) is held. The status is set to \texttt{SKIPPED}, the decision is marked as \texttt{need approval}, and an approval card is rendered. If the user approves, the tool executes; otherwise, it remains blocked.
        \item \textbf{\texttt{NONE / LOW / MEDIUM} Risk}: The action (e.g., \texttt{read\_recent\_emails}) is auto-approved and sent to the tool executor.
    \end{itemize}
    \item \textbf{Post-Execution Synthesis}: The tool output is sent back to the LLM to generate a user-friendly summary.
    \item \textbf{Logging and Rendering}: The event details are written to the database, and the UI displays the response alongside the execution trace.
\end{enumerate}

\section{Key Component Explanations}

\subsection{Groq Cloud \& Llama 3.3 70B Versatile}
Groq utilizes a unique LPU (Language Processing Unit) architecture that provides high-throughput token generation for large language models, significantly reducing inference latency. In the proposed system, Llama 3.3 70B performs intent classification, argument parsing, and final synthesis. The speed of the LPU allows Iris to execute sequential LLM requests while keeping user interface latency under one second.

\subsection{LangChain Framework}
LangChain is used to manage prompts, construct schemas, and handle tool-binding contracts with the language model. We use the \texttt{@tool} decorator to dynamically export Python functions as JSON schemas, which are passed to the model during invocation.

\subsection{OAuth 2.0 \& Gmail REST Integration}
For mail utilities, Iris integrates with Google Gmail APIs using OAuth 2.0 authorization codes. Scoped credentials are stored in \texttt{token.json} to authorize requests without requiring direct password storage.

\subsection{Controlled Intent Routing Pattern}
This architecture replaces the open-loop ReAct mechanism. By forcing a single-turn routing pattern, the LLM acts as an intent classifier and response generator, but has no control over the execution loop itself. This design eliminates looping errors and ensures all actions are intercepted by the governance engine.

\section{Implementation Details}

\subsection{Audit Logging System (\texttt{audit\_logger.py})}
The system implements a triple-logging strategy, writing events to JSON, CSV, and SQLite formats:
\begin{itemize}
    \item \textbf{Unique Session Tracking}: Generates a process-level session token during load: \texttt{SESSION\_ID = str(uuid.uuid4())[:8].upper()}.
    \item \textbf{Thread Safety}: Employs a \texttt{threading.Lock()} mutex lock around all memory buffer updates and file writes, ensuring background tasks do not corrupt logs.
    \item \textbf{Logging Schemas}: Captures the unique event ID, session ID, timestamp, intent, tool arguments, risk evaluation, decision, status, policy reasoning, and output payload.
\end{itemize}

\subsection{Controlled Router (\texttt{agent/controller.py})}
Implements the main execution loop. It validates user queries, extracts arguments, and matches proposed tools against the risk classifier. Depending on the decision, it skips execution, pauses for approval, or executes the tool.

\subsection{Risk Classifier \& Governance Engine}
The risk classifier reads actions from \texttt{risk\_classifier.py} and maps them to risk levels. The decision engine evaluates the risk and applies system policies: auto-approving low-risk tasks, holding high-risk tasks for user confirmation, and systematically blocking critical-risk tasks.

\subsection{Query Validator (\texttt{agent/query\_validator.py})}
Analyzes input query text using regex pattern maps to catch typing mistakes in commands and email domain strings, outputting correction suggestions to the UI.

\subsection{Daemon Task Scheduler (\texttt{tools/scheduler\_tool.py})}
Queues tasks for asynchronous execution by spawning background threads:
\begin{lstlisting}[language=Python]
# Spawns a non-blocking daemon thread
thread = threading.Thread(target=sleep_and_execute, args=(delay, task))
thread.daemon = True
thread.start()
\end{lstlisting}
The background execution runs the tool, evaluates the outcome, and logs the execution back to the main database, using the parent trigger's trace ID.

\section{Algorithms and Mathematical Formulation}

\subsection{Mathematical Model of Risk Policy}
Let $A$ be the set of all possible agent actions, and $R$ be the set of security risk tiers:
\begin{equation}
R = \{\text{NONE}, \text{LOW}, \text{MEDIUM}, \text{HIGH}, \text{CRITICAL}\}
\end{equation}
The risk classification function $f: A \to R$ maps a proposed action $a \in A$ to a risk level:
\begin{equation}
f(a) = \text{classify\_risk}(a)
\end{equation}
The policy decision function $g: R \to D$, where $D = \{\text{approved}, \text{need approval}, \text{reject}\}$, determines the execution path:
\begin{equation}
g(r) = \begin{cases} 
\text{reject} & \text{if } r = \text{CRITICAL} \\
\text{need approval} & \text{if } r = \text{HIGH} \\
\text{approved} & \text{if } r \in \{\text{NONE}, \text{LOW}, \text{MEDIUM}\} 
\end{cases}
\end{equation}
Thus, the final composition function $h: A \to D$ evaluates the action before execution:
\begin{equation}
h(a) = (g \circ f)(a) = g(f(a))
\end{equation}
The actual status of the tool execution is represented by $S = \{\text{SUCCESS}, \text{FAILED}, \text{SKIPPED}\}$:
\begin{equation}
\text{status}(a) = \begin{cases}
\text{SKIPPED} & \text{if } h(a) \in \{\text{reject}, \text{need approval}\} \\
\text{SUCCESS} & \text{if } h(a) = \text{approved} \land \text{execute}(a) \text{ returns without error} \\
\text{FAILED} & \text{if } h(a) = \text{approved} \land \text{execute}(a) \text{ raises an exception}
\end{cases}
\end{equation}

\subsection{Time Complexity of Typo Check Algorithm}
The validator performs a sequential regex-based scan over the input query:
\begin{equation}
T(n) = \sum_{i=1}^{P} C_i \cdot M_i(n)
\end{equation}
where $P$ is the number of defined regex patterns, $C_i$ is the compiled regex matching overhead, and $M_i(n)$ is the complexity of matching pattern $i$ against a query of length $n$. For simple patterns, this resolves to linear time complexity:
\begin{equation}
\mathcal{O}(P \cdot n)
\end{equation}
Because $P$ is small and constant, the runtime remains under one millisecond, preventing input bottlenecks.

\section{Conclusion and Future Work}
This paper presented Iris, a governance framework that addresses the safety and predictability risks of autonomous AI agents. By replacing open-loop ReAct reasoning with a deterministic Controlled Intent Router and applying a multi-format logging pipeline, Iris ensures predictable execution and auditability.

Future work will focus on:
\begin{enumerate}
    \item \textbf{Containerized Sandboxing}: Executing system tools inside isolated Docker containers to protect the host system from dangerous actions.
    \item \textbf{Role-Based Access Control}: Restricting manual approval rights to authorized administrator accounts in the user interface.
    \item \textbf{Semantic Typo Analysis}: Integrating semantic vector matching to detect typos and bypass attempts in user queries.
\end{enumerate}
