# Iris Live Demo Script & Walkthrough Guide
**Milestone: Core Security Governance & Observability Demo**

This guide provides a step-by-step script for demonstrating the Iris security governance layer, typo checker, interactive approval cards, and audit log telemetry dashboard.

---

## 🛠️ Step 0: Launching the App
Before starting the presentation, launch the Streamlit server:
1. Double-click the **`run_iris.bat`** file in the project root directory.
2. The terminal will initialize the virtual environment and open the dashboard in your default browser at `http://localhost:8501`.

---

## 🟢 Scenario 1: Safe Agent Action (Auto-Approved)
*Demonstrates a low-risk, read-only action that is allowed to execute automatically.*

### 1. Action Steps:
* In the chat input, type:
  ```
  read my recent emails
  ```
* Press **Enter**.

### 2. What to explain during the demo:
* **Controlled Intent Router**: The agent didn't enter a loop or ask recursive questions. It extracted the read intent in a single turn.
* **Auto-Approval**: Since reading emails is mapped to `LOW` risk, the system automatically authorized the tool execution.
* **Reasoning Explanation**: Point to the **Execution Trace** block in the chat. Note the explanation:
  > *"This is a read-only operation and does not modify external systems."*
* **Audit Timeline**: Point to the right sidebar (or main timeline). Show the new log entry with a green **`LOW`** badge and **`approved`** status.

---

## 🟡 Scenario 2: Pre-Execution Warning (Typo Checking)
*Demonstrates how Iris safeguards the user from accidental errors (like email typos) before sending requests to the LLM.*

### 1. Action Steps:
* Type the following query with a deliberate typo:
  ```
  send a report to sifatahmed2523@gmal.com
  ```
* Press **Enter**.

### 2. What to explain during the demo:
* **Static Input Validation**: The pre-execution typo checker (`agent/query_validator.py`) matches inputs against known patterns (e.g. `@gmal.com`, `@gmailcom`, `delte`).
* **Instant Warning**: The dashboard displays a warning banner:
  > ⚠️ *It looks like you wrote `sifatahmed2523@gmal.com`. Did you mean `sifatahmed2523@gmail.com`?*
* **Security Justification**: Explain that checking spelling errors first prevents executing actions with wrong parameters (which might send confidential reports to non-existent or malicious domains).

---

## 🔴 Scenario 3: Risky Agent Action (Human-in-the-Loop Approval)
*Demonstrates how high-risk outbound actions are intercepted, held, and require explicit human authorization to run.*

### 1. Action Steps:
* Type a command to send an email:
  ```
  send an email to sifatahmed2523@gmail.com with subject "Status Update" and body "The task is complete."
  ```
* Press **Enter**.

### 2. What to explain during the demo:
* **Policy Interception**: The `risk_classifier.py` mapped `send_email` to `HIGH` risk.
* **Gated State**: The system automatically halted execution, returning a **Pending Approval Card** in the chat feed instead of sending the email.
* **Interactive Decider**: Point to the **Explanation**:
  > *"External communication can affect third parties and therefore requires additional approval."*
* **Approve & Run**: Click the **"Approve & Execute"** button inside the card.
* **Outcome**: The agent completes the OAuth transaction, sends the email, displays the success message, and updates the timeline log status to **`SUCCESS`**.

---

## 💀 Scenario 4: Blocked Action (Critical Risk)
*Demonstrates how dangerous administrative or scripting actions are systematically blocked to prevent system takeover.*

### 1. Action Steps:
* Type a query trying to run code or system commands:
  ```
  run a terminal command 'echo hello' to check the folder files
  ```
* Press **Enter**.

### 2. What to explain during the demo:
* **Strict Security Boundary**: Running arbitrary shell commands or executing python files is mapped to `CRITICAL` risk.
* **Automatic Rejection**: The decision engine blocks the tool execution immediately. No button or bypass is available to the user.
* **Block Notification**: The dashboard renders a dark red card displaying:
  > ❌ *This action has been rejected because it is classified as critical and violates strict security boundaries.*
* **Audit Logging**: Show that a **`CRITICAL`** / **`reject`** / **`SKIPPED`** entry has been appended to the SQLite and JSON database logs, providing permanent compliance records of the blocked attempt.

---

## 📊 Scenario 5: Dashboard Timeline & Telemetry Observability
*Demonstrates how administrators can audit, search, and export logs generated by the agent.*

### 1. Action Steps:
* Navigate to the **"Audit Log"** tab at the top of the dashboard.

### 2. What to explain during the demo:
* **Session Summary Metrics**: Point to the metric cards showing:
  * Total Actions Logged
  * Unique Tools Invoked
  * Execution Success Rate
  * Interactive Risk Distribution chart
* **Unified Audit Database**: Show that logs can be filtered dynamically by risk tier (Low, Medium, High, Critical) and text search.
* **Log Data Portability**: Point to the **"Download JSON Logs"** button, proving that the full audit database is structured, portable, and ready for SOC compliance ingestion.
